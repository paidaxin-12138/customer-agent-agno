# 拼多多客服：退换货/售后卡片发送可行性调研

> **文档性质**：技术调研（不改代码）  
> **适用项目**：Customer-Agent（拼多多 WebSocket + MMS HTTP + Agno）  
> **更新说明**：基于仓库 `main` 分支与公开/社区逆向信息整理

---

## 1. 调研结论（摘要）

| 问题 | 结论 |
|------|------|
| **当前软件能否发退换货卡片？** | **不能**。出站仅支持文本、图片、**商品卡片**。 |
| **有没有实现可能？** | **有，中等偏高**，但依赖 **MMS 非公开接口** + 本地抓包验证，不是开放平台正式能力。 |
| **开放平台售后 API 能否发卡？** | **不能主动发聊天卡**；适合处理买家**已提交**的售后单（同意/拒绝/查询）。 |
| **零风险兜底方案** | **文字引导**买家到订单页申请售后（项目已支持）。 |

---

## 2. 先分清「退换货卡片」的四种含义

| 类型 | 典型场景 | Customer-Agent 现状 |
|------|----------|---------------------|
| **A. 买家发来的订单/售后卡片** | 买家从订单进聊天，带售后状态 | **能收**（`sub_type=1` → `ORDER_INFO`），**不能主动发** |
| **B. 商家主动推「申请退款/售后」** | 客服工作台「帮买家发起售后」 | **未实现**；社区有 MMS 接口线索 |
| **C. 处理买家已提交的售后单** | 后台同意/拒绝退货退款 | `after_sales.py` 有示例，**未接入消息链/AI** |
| **D. 文字引导退换货** | 「请到订单里申请售后」 | **已支持**（关键词、知识库、AI 话术） |

本文重点分析 **类型 B**（商家在聊天里主动发售后/退款类结构化消息）。

## 3. 本仓库当前出站能力

### 3.1 已实现（`SendMessage`）

文件：`Channel/pinduoduo/utils/API/send_message.py`

| 方法 | MMS 地址 | 说明 |
|------|----------|------|
| `send_text` | `plateau/chat/send_message` | 纯文本 |
| `send_image` | `plateau/chat/send_message` | 图片 |
| `send_mallGoodsCard` | `plateau/message/send/mallGoodsCard` | **商品卡片** |
| `move_conversation` | `plateau/chat/move_conversation` | 转接会话 |

Agno 工具：`send_goods_link` → 内部调用 `send_mallGoodsCard`（`Agent/CustomerAgent/tools/send_goods_link.py`）。

实时聊天 UI：`ui/chat_ui.py` 中「发商品卡」按钮，同样走 `send_mallGoodsCard`。

### 3.2 未实现

- 退换货 / 售后 / 订单类**商家主动卡片**
- `ask_refund_apply`、`sendUserHelpLink` 等 MMS 接口
- 任何 Agno Tool 用于售后发卡

### 3.3  inbound（能收什么）

`Channel/pinduoduo/pdd_message.py` 在 `user_msg_type == 0` 且 `sub_type == 1` 时解析**订单信息**，含：

```python
order_info = {
    "order_id": ...,
    "goods_id": ...,
    "goods_name": ...,
    "afterSalesStatus": ...,  # 售后状态
    "afterSalesType": ...,    # 售后类型
    "spec": ...,
}
```

即：买家发订单卡 → 系统可识别；**不等于**商家能回发同类卡片。

## 4. 开放平台路线（`after_sales.py`）

文件：`Channel/pinduoduo/utils/API/after_sales.py`  
网关：`gw-api.pinduoduo.com/api/router`（`OpenPlatformAPI`）

| 接口 type | 用途 |
|-----------|------|
| `pdd.refund.returngoods.agree` | 同意退货退款 |
| `pdd.refund.returngoods.reject` | 拒绝退货退款 |
| `pdd.after.sale.status.get` | 查询售后单状态 |
| `pdd.after.sale.list.order.get` | 售后单列表 |
| `pdd.after.sale.limit.info.get` | 售后限额 |

### 4.1 能做什么

- 买家**已在平台提交售后**后，商家自动/半自动**同意、拒绝、查询**。
- 需在 `config.json` 配置 `pinduoduo_open`（`client_id`、`client_secret`、`access_token`）。

### 4.2 不能做什么

- **不能**在聊天窗口里主动插入一张「请申请退换货」的**平台卡片**。
- 买家聊天里的售后相关展示，多由**平台在买家申请后推送**，不是 Router 接口直接「发卡」。

### 4.3 与本项目关系

- 代码为**示例/预留**，**未**挂到 `handler_chain` 或 `CustomerAgent` tools。
- 与 `send_mallGoodsCard` **互补**：一个管「后台售后单」，一个管「聊天结构化消息」。

## 5. MMS 商家聊天路线（最有希望的扩展方向）

与商品卡相同：**登录 Cookie + `anti-content` + `mms.pinduoduo.com`**，属于商家后台 Web 内部接口，**无拼多多开放平台正式文档**。

### 5.1 本仓库已用的商品卡（对照）

```http
POST https://mms.pinduoduo.com/plateau/message/send/mallGoodsCard
Content-Type: application/json

{
  "uid": "<买家 UID>",
  "goods_id": "<商品 ID>",
  "biz_type": 2
}
```

实现见 `send_message.send_mallGoodsCard`。

### 5.2 社区逆向发现的售后相关接口

来源：[jiulingyun/pdd-customer-service-sdk](https://github.com/jiulingyun/pdd-customer-service-sdk)（Playwright + 抓包封装，**非官方**）。

| 接口（逆向） | 推断用途 | 主要参数（模型定义） |
|--------------|----------|----------------------|
| `POST .../plateau/message/ask_refund_apply/send` | 商家在会话内向买家**发起退款/售后申请** | `order_sn`, `after_sales_type`, `question_type`, `refund_amount`, `message` |
| `POST .../latitude/message/sendUserHelpLink` | 发送**帮助/引导链接**（可能与订单/售后入口相关） | `link_type`, `uid`, `order_sn` |

对应 Pydantic 模型（SDK `models/api_request_model.py`）：

```python
class AskRefundApplyRequest(BaseModel):
    order_sn: str
    after_sales_type: int
    question_type: int
    refund_amount: int
    message: str = None

class SendUserHelpLinkRequest(BaseModel):
    link_type: int
    uid: str
    order_sn: str
```

**注意**：`after_sales_type`、`question_type`、`link_type` 的枚举值需通过**商家后台抓包**确认，社区仓库未保证与线上一致。

### 5.3 与「退换货卡片」的对应关系（待验证）

| 你在后台看到的操作 | 可能对应的请求 | 验证状态 |
|--------------------|----------------|----------|
| 发送「申请退款」/ 帮买家填售后 | `ask_refund_apply/send` | 社区有实现，**本项目未验证** |
| 发送订单/售后帮助入口 | `sendUserHelpLink` | 需确认 `link_type` |
| 发送商品推荐卡 | `send/mallGoodsCard` | **本项目已用** |

## 6. 实现路径对比

```text
┌─────────────────────────────────────────────────────────────────┐
│ 路径 1：MMS Cookie HTTP（推荐优先验证）                            │
│   抓包 → 封装 SendMessage → 可选 Agno Tool                        │
│   难度：中 | 与商品卡同架构 | 风险：非官方、改版、风控              │
├─────────────────────────────────────────────────────────────────┤
│ 路径 2：开放平台 after_sales（仓库已有 after_sales.py）          │
│   买家已申请售后 → 自动同意/拒绝/查询                             │
│   难度：中 | 不解决「主动发卡」| 需 pinduoduo_open                 │
├─────────────────────────────────────────────────────────────────┤
│ 路径 3：Playwright 点 UI（社区 SDK 部分能力）                     │
│   难度：高 | 维护成本高 | 与当前 WebSocket+HTTP 架构不一致         │
├─────────────────────────────────────────────────────────────────┤
│ 路径 4：仅文字引导（现状）                                        │
│   难度：无 | 已支持 | 无平台卡片体验                              │
└─────────────────────────────────────────────────────────────────┘
```

### 6.1 若采用路径 1，建议工程步骤（规划用）

1. **抓包**：登录 `https://mms.pinduoduo.com/chat-merchant/`，手动发一次目标卡片，记录 URL、Body、Headers（含 `anti-content`）。
2. **对齐**：确认是 `ask_refund_apply/send` 还是 `sendUserHelpLink` 或其它路径。
3. **封装**：在 `SendMessage` 增加方法（仿 `send_mallGoodsCard`），复用 `BaseRequest` Cookie 与重试。
4. **参数**：`order_sn` 来自会话上下文或订单查询 API；金额/类型需业务规则 + 抓包枚举。
5. **接入**：可选 `Agent` Tool + 指令约束（何时允许代买家发起售后）。
6. **测试**：用小号订单验证买家端展示与风控。

### 6.2 若采用路径 2，建议工程步骤（规划用）

1. 设置页已支持 `pinduoduo_open` 时，在 `OrderLogisticsHandler` 或新 Handler 中调用 `AfterSalesManager`。
2. 需买家消息或订单上下文带出 `after_sale_id`（往往要先 list 再 agree）。
3. **不替代**聊天发卡，仅减少人工点后台。

## 7. 本地抓包操作指南（不改代码即可做）

1. Chrome 打开商家客服：`chat-merchant/index.html`，确保已登录。
2. F12 → **Network**，筛选：`plateau`、`latitude`、`refund`、`after`、`send`。
3. 在聊天工具栏找到「申请退款」「发送售后」「帮助链接」等按钮（以你店铺实际 UI 为准），点击一次。
4. 记录：
   - Request URL
   - Request Payload（JSON）
   - Response
   - 是否依赖 `anti-content` 请求头
5. 在买家 App 侧确认是否出现**卡片样式**（非纯文字）。

若**后台无「发给买家的退换货卡」按钮**，则软件只能：

- 文字引导（路径 4）+ 开放平台处理已有售后单（路径 2）。

## 8. 能力矩阵（决策表）

| 能力 | 当前软件 | 实现可能性 | 依赖 |
|------|----------|------------|------|
| 发**商品卡片** | ✅ | — | Cookie、`send_mallGoodsCard` |
| 发**退换货/申请退款聊天卡** | ❌ | **中高** | MMS 逆向 + 抓包 + `order_sn` 等 |
| 发**订单卡片**（商家主动） | ❌ | **待验证** | 需抓包 |
| **同意/拒绝**已有售后 | 代码有、未接线 | **中高** | 开放平台授权 |
| **文字**引导退换货 | ✅ | **已实现** | 无 |

## 9. 风险与合规提示

1. **非公开接口**：`plateau/message/*`、`latitude/message/*` 可能随商家后台改版失效，需做版本监控。
2. **风控**：高频自动发起退款/售后可能触发平台限制，需频率控制与人工审核策略。
3. **合规**：社区 SDK 均标注仅供学习；商用需自行评估拼多多服务条款。
4. **误操作**：AI 自动 `ask_refund_apply` 涉及金额与售后类型，必须有确认策略或仅人工触发。

## 10. 与本项目其它模块的关系

| 模块 | 关系 |

| `OrderLogisticsHandler` | 改地址转人工；物流查轨迹；**不**发售后卡 |

| `AIReplyHandler` | 出站仅 `send_text`；无售后卡 Tool |
