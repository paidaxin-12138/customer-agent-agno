# Customer-Agent 生产部署说明

> **重要**：本软件需要图形界面环境（Windows / macOS / Linux with X11）。无显示器或纯 SSH 环境下必须使用 `xvfb-run` 模拟显示，否则 PyQt6 无法启动。
>
> **Cookie 有效期**：拼多多商家后台 Cookie 通常 **7–30 天** 失效。请定期在「账号管理」确认店铺为**在线**状态；失效后需重新扫码/登录（本应用**不会**自动续期 Cookie，仅会通知并尝试刷新）。

面向 **每天咨询 < 100 人** 的无人值守本地部署：PM2 守护、健康检查、自动备份与数据清理。

## 目录约定

| 路径 | 说明 |
|------|------|
| `data/customer_agent.db` | 主 SQLite（`config.json` → `db_path`） |
| `logs/` | `out.log` / `error.log` / `boot.log` / `app_debug.log`（按日+按大小轮转） |
| `backup/` | 每日备份 `customer_agent_YYYY-MM-DD.db` |
| `temp/` | 临时文件（默认 7 天清理） |

兼容旧库：若仍使用 `temp/customer.db`，备份脚本会自动探测。

## 1. PM2 守护

```bash
cd /path/to/Customer-Agent-main
uv sync
npm i -g pm2
pm2 start ecosystem.config.js
pm2 save
pm2 startup   # 按提示配置开机自启
```

日志：`logs/out.log`（INFO+）、`logs/error.log`（ERROR+）、`logs/boot.log`（启动诊断与未捕获异常）。

**注意**：PyQt6 需本机图形会话；Linux 服务器请用桌面或 `xvfb-run`。

### PM2 日志轮转

应用内 `utils/logging_setup.py` 已对 `out.log` / `error.log` 做**按日轮转、保留 7 天**（与 PM2 写入同一文件）。

若还需轮转 PM2 **进程管理器自身**的日志缓存，可安装官方插件：

```bash
pm2 install pm2-logrotate
pm2 set pm2-logrotate:max_size 10M
pm2 set pm2-logrotate:retain 7
pm2 set pm2-logrotate:compress true
```

### 常用 PM2 命令

```bash
pm2 status customer-agent    # 查看状态
pm2 restart customer-agent   # 重启
pm2 logs customer-agent      # 实时日志
pm2 stop customer-agent      # 停止
```

## 2. 健康检查

配置项：`production.health_host`（默认 `127.0.0.1`）、`production.health_port`（默认 `8080`）。

### `/health` — 存活探针（Liveness）

进程内 HTTP 服务正常即返回 200，**不检查** WebSocket 或消费者状态。

```bash
curl -s http://127.0.0.1:8080/health
# {"status":"ok","timestamp":1717000000}
```

用途：PM2 / Docker / systemd 判断进程是否存活、是否需重启。

### `/ready` — 就绪探针（Readiness）

检查是否**至少有一个店铺 WebSocket 已连接**，且对应 `pdd_{shop_id}` 消息消费者正在运行。

```bash
curl -s http://127.0.0.1:8080/ready
# 就绪示例（200）：
# {"ready":true,"timestamp":...,"ws_connected":1,"ws_total":1,"consumers_running":[...]}
#
# 未就绪示例（503）：
# {"ready":false,"reason":"no_websocket_connected","ws_connected":0,...}
```

| 探针 | 失败含义 | 建议动作 |
|------|----------|----------|
| `/health` 不可达 | 进程崩溃或未启动 | `pm2 restart` / 查 `logs/boot.log` |
| `/health` 正常但 `/ready` 503 | 进程活着但未连上拼多多 | 查 Cookie、账号在线状态、WS 日志 |
| 两者均正常 | 可正常收发消息 | — |

### `/metrics`

```bash
curl -s http://127.0.0.1:8080/metrics
# 消息处理数、队列深度、运行时长等
```

## 3. 数据库备份

```bash
uv run python scripts/backup_db.py
```

定时：每日 `production.backup_hour:backup_minute`（默认 02:00），保留 `backup_retention_days` 天。

## 4. 生命周期清理

每日 `retention.lifecycle_hour`（默认 03:00）：

- 删除超期 `chat_messages`
- 删除超期 `ops_security_audits`
- 清理 `retention.temp_dir` 下旧文件
- 按 `vacuum_interval_days` 执行 `VACUUM`
- （可选）`retention.vector_days > 0` 时清理超期 LanceDB / `knowledge_docs.json` 向量文档

## 5. WebSocket 重连

`chat.ws_reconnect_delay_sec`（默认 5s）、`ws_reconnect_max_attempts`（0=无限）。  
认证失败、断线后自动重连；**消息消费者保持运行**。

## 6. 审计日志

`utils/audit_log.audit_log()` → `ops_security_audits`。  
已接入：登录、关键词增删改、退款卡发送、**改址弹窗推送**、配置保存、系统启停、Cookie 过期。

## 7. Docker（可选）

```bash
docker compose up -d --build
```

挂载 `./data`、`./logs`、`./config.json`。

**健康检查**：Dockerfile / compose 内 `healthcheck` 在**容器内**访问 `http://127.0.0.1:8080/health`。  
默认**不**将 8080 映射到宿主机（`docker-compose.yml` 中 `ports` 已注释）。  
若需从宿主机探针，取消 compose 中 `ports: "127.0.0.1:8080:8080"` 并在 `config.json` 设置：

```json
"production": {
  "health_host": "0.0.0.0",
  "health_port": 8080
}
```

## config.json 生产关键项

除 `retention` / `production` 外，建议在 `chat` 段显式配置（默认值见 `config.py` → `config_base`）：

```json
{
  "chat": {
    "ws_reconnect_delay_sec": 5,
    "ws_reconnect_max_attempts": 0,
    "ws_auto_reconnect_enabled": true,
    "ai_mode_check_fail_open": false,
    "ai_mode_check_retries": 3,
    "unhandled_fallback_enabled": true,
    "unhandled_fallback_notice": "亲，消息已收到，客服稍后会回复您；如需人工请回复「人工」。",
    "message_consumer_max_concurrent": 28
  },
  "production": {
    "health_enabled": true,
    "health_host": "127.0.0.1",
    "health_port": 8080,
    "backup_enabled": true,
    "backup_hour": 2,
    "backup_minute": 0,
    "backup_retention_days": 7,
    "log_level": "INFO"
  },
  "retention": {
    "chat_history_days": 30,
    "audit_log_days": 90,
    "temp_files_days": 7,
    "vacuum_interval_days": 30,
    "vector_days": 0
  }
}
```

- `ai_mode_check_fail_open: false`：DB 查询 ai_mode 连续失败时**关闭 AI**（fail-closed，生产推荐）。
- `ws_reconnect_max_attempts: 0`：WebSocket **无限重连**。
- `retention.vector_days`：`0` 表示不清理 LanceDB 向量。

环境变量（优先于 config.json）：

| 变量 | 说明 |
|------|------|
| `LLM_API_KEY` | LLM API 密钥 |
| `AGENT_CREDENTIAL_KEY` | Cookie/密码加密主密钥 |
| `STRICT_CONFIG=1` | 启动配置缺失时硬失败 |

## 9. Windows 服务化（NSSM 备选）

适用于需要开机自启、但不想使用 PM2 的 Windows 环境。

### 前置条件

1. 安装 [NSSM](https://nssm.cc/download) 并将 `nssm.exe` 加入 PATH。
2. 在项目根目录执行 `uv sync`，确保存在 `.venv\Scripts\python.exe`。
3. 配置环境变量（系统或 NSSM 服务环境）：
   - `AGENT_CREDENTIAL_KEY`（凭据加密）
   - `LLM_API_KEY`（推荐，避免在 config.json 明文保存）
4. 本机已登录图形桌面会话（服务账户需能访问显示器，或使用允许交互式桌面的方案）。

### 注册 / 卸载

以**管理员**运行：

```bat
scripts\install_windows_service.bat
scripts\uninstall_windows_service.bat
```

服务名：`CustomerAgent`。注册后：

```bat
nssm start CustomerAgent
nssm status CustomerAgent
```

日志：`logs\service-out.log`、`logs\service-error.log`（由 NSSM 捕获 stdout/stderr）。

## 10. 故障排查

### Cookie 过期

**现象**：账号显示离线；日志出现 `session_expired`；桌面通知「拼多多客服 - Cookie 过期」；MMS 接口返回登录失效。

**处理**：

1. 打开应用 → **账号管理** → 对应店铺 **重新扫码/登录**。
2. 确认系统托盘/桌面通知权限已开启（macOS 需在「系统设置 → 通知」中允许）。
3. 审计表 `ops_security_audits` 中可检索 `session_expired` 事件。

> 本应用**不会**自动续期 Cookie，仅通知并尝试 Playwright 刷新（需图形环境）。

### `/ready` 返回 503

**排查步骤**：

1. `curl http://127.0.0.1:8080/ready` 查看 `reason` 字段：
   - `no_connection_registered`：尚未添加/启动任何账号。
   - `no_websocket_connected`：WS 断线或 Cookie 失效 → 查账号在线状态。
   - `no_running_consumer_for_connected_shop`：消费者未启动 → 重启应用或查 `logs/error.log`。
2. 应用内查看店铺连接状态（在线/重连中/断开）。
3. 查 `logs/out.log` 中 `WebSocket 重连` / `Consumer` 相关日志。

### 消息无回复

1. 确认 `/ready` 为 200。
2. 查 `LLM_API_KEY` 是否配置；`STRICT_CONFIG=1` 时缺密钥会启动失败。
3. 查 `logs/error.log` 中 Handler / LLM 超时错误。
4. 确认会话未处于人工模式（`ai_mode=false`）。

### 日志位置

| 文件 | 内容 |
|------|------|
| `logs/out.log` | INFO 及以上（按日轮转，7 天） |
| `logs/error.log` | ERROR 及以上 |
| `logs/boot.log` | 启动诊断、未捕获异常、faulthandler 崩溃栈 |
| `logs/app.log` | 全级别（按大小轮转） |
| `logs/app_debug.log` | DEBUG（仅 `production.log_level=DEBUG` 时） |

查看示例：

```bash
tail -f logs/out.log
tail -100 logs/error.log
grep -i "websocket\|cookie\|handler" logs/out.log | tail -20
```

### PM2 反复重启

```bash
pm2 logs customer-agent --lines 100
cat logs/boot.log
```

常见原因：无图形环境（PyQt6 启动失败）、配置硬失败（`STRICT_CONFIG=1`）、依赖缺失。  
PM2 在 10s 内崩溃超过 50 次会停止自动重启，需 `pm2 restart customer-agent` 手动恢复。
